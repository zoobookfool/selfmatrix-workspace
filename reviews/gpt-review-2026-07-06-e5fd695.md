## Findings

### [P1] Route A generated nginx still proxies Synapse to `8008` instead of the published `8028`

Home-side Route A publishes Synapse as `${BACKEND_BIND_IP}:8028 -> 8008`, and the comments tell the VPS edge to proxy Matrix to home `:8028`. However `scripts/provision-rtc-vps.sh --with-edge` still generates `proxy_pass http://${HOME_BACKEND_IP}:8008`.

Evidence:

- `docker-compose.route-a.example.yml:15` says Synapse is exposed as `${BACKEND_BIND_IP}:8028 -> 8008`.
- `docker-compose.route-a.example.yml:18` says the VPS edge should proxy `matrix.example.com -> :8028`.
- `docker-compose.route-a.example.yml:29` maps `${BACKEND_BIND_IP}:8028:8008`.
- `scripts/provision-rtc-vps.sh:256` generates `proxy_pass http://${HOME_BACKEND_IP}:8008`.

Dry-run output confirmed:

```text
server_name matrix.example.com;
location / {
    proxy_pass http://100.64.0.1:8008;
}
```

Impact: the documented Route A provisioning path leaves `matrix.example.com` pointing at a port that is not exposed by the home-side override, so Matrix API/federation traffic fails.

Suggested fix: generate `proxy_pass http://${HOME_BACKEND_IP}:8028;` or introduce a single `HOME_SYNAPSE_PORT=8028` variable used by both docs and generated nginx.

### [P1] Generated nginx edge still exposes `/_synapse/admin`

The Caddyfile blocks the Synapse Admin API, and `docs/operations.md` says public edge proxies must deny `/_synapse/admin`. The generated nginx config does not include that deny block; it has only a catch-all Matrix `location /`.

Evidence:

- `caddy/Caddyfile:23` blocks `/_synapse/admin*`.
- `docs/operations.md:119` instructs edge proxies to block `/_synapse/admin`.
- `docs/operations.md:126` shows an nginx deny block.
- `scripts/provision-rtc-vps.sh:252` starts a catch-all `location /` for `MATRIX_HOST`, with no preceding admin block.

Impact: users following `--with-edge` can accidentally expose Synapse Admin API routes to the Internet. The API still needs credentials, but this violates the repo's stated public-edge security model.

Suggested fix: add this before `location /` in the generated `MATRIX_HOST` server:

```nginx
location ^~ /_synapse/admin {
    return 403;
}
```

### [P1] Generated nginx still creates the well-known directory but never writes Matrix well-known JSON

Route A disables the home-side Caddy, so the VPS edge must serve `/.well-known/matrix/server` and `/.well-known/matrix/client`. The generated nginx config only sets a static root and creates the directory; it does not create the required JSON files.

Evidence:

- `caddy/Caddyfile:9` and `caddy/Caddyfile:10` serve the required inline well-known JSON in the default deployment.
- `scripts/provision-rtc-vps.sh:241` defines a `location /.well-known/matrix/` with `root /var/www/selfmatrix-well-known`.
- `scripts/provision-rtc-vps.sh:300` only creates `/var/www/selfmatrix-well-known/.well-known/matrix`.
- No command writes `server` or `client` files under that directory.

Impact: Matrix discovery fails for Route A deployments unless the operator manually creates files the script does not create.

Suggested fix: either generate the two files or use nginx `return 200` blocks. The generated client well-known should optionally include `org.matrix.msc4143.rtc_foci` when the RTC host is being configured.

### [P2] New SFU selection correction did not update `docs/architecture.md`

The latest commit corrected `docs/requirements.md` to say MatrixRTC focus selection is first-session-participant / oldest participant behavior, and that users on other homeservers can technically be first participant. However `docs/architecture.md` still says the call is hosted by "the participant who started it" and that users without their own server can only join, not start.

Evidence:

- `docs/requirements.md:68` says the first participant's SFU is selected.
- `docs/requirements.md:69` says other-homeserver-only users can technically start by joining first.
- `README.md:104` repeats the first-participant / first-come behavior.
- `docs/architecture.md:105` still contains the old "started participant's SFU" and "join only" wording.

Impact: design docs now contradict each other on an important operational rule. Operators reading architecture may make the wrong moderation/ordering assumption for calls.

Suggested fix: update `docs/architecture.md:105` to match `docs/requirements.md:68-69`.

### [P2] `.env` is still sourced as shell code in sensitive scripts

`generate-synapse-config.sh`, `backup.sh`, and `restore.sh` still use `source` for `.env`. This treats secrets as shell code, not data. A generated password containing `$`, command substitution, quotes, backticks, or similar characters can be expanded, altered, or break scripts. `generate-synapse-config.sh` then injects the PostgreSQL password into YAML using manual f-string formatting.

Evidence:

- `scripts/generate-synapse-config.sh:21` sources `.env`.
- `scripts/backup.sh:16` sources `.env`.
- `scripts/restore.sh:30` sources `.env`.
- `scripts/generate-synapse-config.sh:67` writes `POSTGRES_PASSWORD` into YAML via string interpolation.

Impact: legal random secret values can generate invalid `homeserver.yaml` or create mismatched DB credentials. If `.env` is not fully trusted, sourcing it also creates code-execution risk.

Suggested fix: parse `.env` as data and emit YAML with a YAML library. The repo already uses non-sourcing patterns in `generate-cinny-config.sh` and `generate-synapse-admin-config.sh`.

### [P2] Backup artifacts containing secrets still rely on default umask and plain off-host sync

Backups include DB dumps, `homeserver.yaml`, Synapse signing keys, and media. The backup directory is created with `mkdir -p`, then ownership is changed to the operator, but permissions are not tightened. Documentation recommends plain `rclone sync backups/ ...`.

Evidence:

- `scripts/backup.sh:22` creates the output directory without explicit mode.
- `scripts/backup.sh:33` archives `homeserver.yaml` and signing keys.
- `scripts/backup.sh:47` changes owner but does not `chmod`.
- `docs/operations.md:158` recommends `rclone sync backups/ remote:selfmatrix-backups`.

Impact: with a permissive umask, local users may read secrets. Off-host backup storage may receive plaintext DB/config/signing-key material.

Suggested fix: add `umask 077`, create backup directories as `0700`, enforce `chmod -R go-rwx "$OUT"`, and document encrypted off-host storage such as `rclone crypt`, restic, borg, or age-encrypted archives.

### [P2] `.env.example` still makes Synapse use `latest` by default

`compose.yaml` has a pinned fallback of `v1.155.0`, but the Quick Start copies `.env.example`, and `.env.example` sets `SYNAPSE_TAG=latest`. In practice, default users get `latest`, not the pinned fallback.

Evidence:

- `.env.example:11` sets `SYNAPSE_TAG=latest`.
- `compose.yaml:19` and `compose.yaml:30` use `SYNAPSE_TAG:-v1.155.0`.

Impact: Synapse upgrades can involve database migrations and operational behavior changes. `latest` makes deploys less reproducible and can surprise operators after image pulls.

Suggested fix: set `SYNAPSE_TAG=v1.155.0` in `.env.example`, or remove it from `.env.example` so the Compose fallback applies.

### [P3] CI ShellCheck still scans only `scripts/`, not `rtc/*.sh`

Manual ShellCheck including `rtc/*.sh` passed at warning level, so there is no current failure. But CI's ShellCheck job still scans only `./scripts`.

Evidence:

- `.github/workflows/ci.yml:18` has `scandir: "./scripts"`.
- `rtc/generate-livekit-config.sh` is a shell script and is not covered by that scan.

Suggested fix: configure ShellCheck to include both `scripts` and `rtc`, or add a small explicit shellcheck command in CI for `rtc/*.sh`.
