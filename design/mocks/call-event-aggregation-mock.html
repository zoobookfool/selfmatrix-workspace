§1 現状の洪水 (再現)

実コード確認済み

後追い参加や履歴ロード時、1 回の通話 (出入りあり・3 人) だけでタイムラインがこうなる。 実際のドッグフーディングでは通話が長い/出入りが多いほど際限なく増える。

1 回の通話 = 12 行 (テストユーザー 3 人、21:04–21:47)

文言は cinny 実物 (ja.json "joined_call": " が通話に参加しました" / "ended_call": " が通話を終了しました")。アイコンは Phone / PhoneDown 相当。

原因 (実装確認済み): RoomTimeline.tsx:1472-1522 のレンダラーがイベント 1 件 = 1 行で描画。 間引きは「参加中の更新 (application が前後とも存在)」の 1 パターンのみ。呼び出し元の eventRenderer (1626-1715) が計算している collapsed フラグ (1650-1657、直前と同送信者・同型・2 分以内) は、このレンダラーの引数宣言 (先頭 3 引数のみ) に含まれておらず実質未使用 (呼び出し側は全レンダラーに同じ 5 要素を渡している)。

§2 Discord の見せ方

一部実測・一部宿題

Discord は「通話の出入りをタイムラインに 1 行ずつ流す」ことをそもそもしない。文脈で 2 通り。

(a) サーバーのボイスチャンネル — タイムラインには何も出ない

参加/退出イベントはテキストチャンネルに一切表示されない (0 件)。「誰がいるか」は サイドバーの VC 名の下にライブ表示され、履歴としては残らない。
# general
🔊 Voice Lounge
AUser A
BUser B 🎙🚫
# random

実使用知識ベース。アプリ実測での裏取りは宿題 (下の todo 参照)。

(b) DM / グループ DM — 通話 1 回 = カード 1 件

開始時に 1 件のシステムメッセージが出て、終了後は同じ 1 件が通話時間付きの表示に落ち着く。 途中参加/退出の個別行は出ない。
📞
User A が通話を開始しました
通話時間 43分

実使用知識ベース。正確な文言 (「開始しました/始めました」等) は要実測。

(c) 折りたたみ表示パターン (実測済み — 転用元)

bot コマンド使用履歴の「○○さんが #play を使用しました」は小さい 1 行 + 本体メッセージの 折りたたみ表示 — discord-behavior-log.html §8 で実測済み。 「頻発する低情報イベントは小さく畳む」という Discord の一般パターンとして、通話イベント集約にそのまま転用できる。
宿題 (アプリ実測): (a)(b) は実使用の記憶ベース。Discord デスクトップアプリで ①サーバー VC の出入りがテキストタイムラインに本当に何も出さないか、②DM 通話カードの正確な文言・終了後表示、 を再確認したら本モックの注記を「実測済み」に更新する (discord-behavior-log.html と同じ運用)。

§4 比較と推奨

案洪水解消情報保持Discord パリティ実装コスト
A: 通話セッションカード ○ (通話 1 回 = 1 件) ○ 展開で全イベント + 参加導線も得る DM 通話カード相当 (サーバー VC とは別解) 中 — セッション境界の判定 + カード UI + 展開
B: 折りたたみ 1 行 ○ (連続 n 件 = 1 行) ○ 展開で全イベント bot コマンド折りたたみと同型 (実測済みパターン) 小 — 連続 run のグルーピングのみ
C: 非表示 ◎ (0 件) × 履歴が消える サーバー VC 完全パリティ 最小 — レンダラーで null を返すだけ
推奨: 目標は案 A、最小着地は案 B。backlog P2 の完了条件 (「連続する通話参加/終了イベントが折りたたまれ、 履歴で洪水にならない」) は B で満たせるので、まず B を入れて洪水を止め、A のカード化 (通話中の参加ボタン込み) は 次の一手として backlog に積み直してよい。C は履歴が消えるため非推奨 (Discord 突き合わせは受け入れ条件ではなく参考、 の方針どおりパリティより情報保持を取る)。

§5 実装メモ (GPT 向け)

論点内容
集約の入れ場所 RoomTimeline.tsx の eventRenderer (1626-1715) で org.matrix.msc3401.call.member の連続 run を検出し、run の先頭だけ集約コンポーネントを 描画・残りは null。レンダラー辞書 (useMatrixEventRenderer) は 1 イベント単位なので、 run 判定はループ側で持つ必要がある。1672 以降 (日付区切り / new divider の組み立てと最終 return) も実装時に併読 — テスト観点③⑤の挙動を左右する
既存 collapsed 判定は流用不可 既存フラグ (1650-1657) は「直前と同送信者・同型・2 分以内」。通話イベントは送信者を跨いで集約したいので 条件が合わない。通話用の run 判定 (同型のみ・時間窓は通話セッション単位) を別に書く
セッション境界 (案 A のみ) 「application が現れた最初のイベント〜全員の membership が消えるまで」を 1 セッションとするのが素直。 隣接する別セッションとの間にテキストメッセージが挟まれば自然に分かれる。案 B なら「連続 run = 1 折りたたみ」で足り、 境界問題は発生しない
i18n 新キー (例: room.timeline.call_events_folded 「通話イベント {{count}}件」等) は en / ja 両方に追加 (cinny は完全パリティ運用)。動的キー t(`...${x}`) は禁止
テスト観点 ①後追い参加 (履歴ロード) で洪水しない ②展開で全イベントが出る ③通話イベントの間にテキストが挟まると 折りたたみが分かれる ④「参加中の更新」抑制 (既存の null 返し) と干渉しない ⑤ライブで 1 件ずつ届く場合も 直前の折りたたみに合流する

web 共通の修正なので web 版 (npm start) で再現→修正→確認が最速。native への反映はビルド往復不要 (cinny 本体の変更のため)。
