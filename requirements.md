# Requirements

2026-07 の要件整理の結果です。設計・実装の判断はこのドキュメントを正とします。

タグの意味:

- `MUST`: 初期リリースまでに満たす
- `SHOULD`: 満たしたいが、実装コスト次第で時期を調整できる
- `LATER`: 将来対応。今は設計で道を塞がないことだけ担保する
- `OUT`: 当面対象外と明示的に決めたもの

## 1. ゴール

- Discord の代替となる、自己所有のチャット・通話基盤を作る `MUST`
- Discord への課金なしで、高画質・高音質の配信通話ができる `MUST`
- プラットフォーム側の BAN でチャット・画像などのログが消えない構造にする `MUST`
  - 守る対象は記録(チャット履歴、メディア、誰が居たかの在籍情報)。BAN 等によるコミュニティへのアクセス喪失そのものは許容する
  - E2EE ルームの履歴を「読める状態」で残すには暗号鍵が必要なため、リカバリーキーも保全対象とする(§8 参照)

## 2. テキストチャット

- Discord 風 UI(Space=サーバー、Room=チャンネルの見せ方) `MUST`
- E2EE ルームを標準運用にする `MUST`
- 内容は既存の docs/architecture.md の方針を踏襲する

## 3. 通話・配信(WebRTC / MatrixRTC 系統)

- グループ通話。想定同時人数は 10 人前後 `MUST`
- 通話は音声 + 画面共有を基本とし、カメラ映像は扱わない `OUT`
- 通話参加者それぞれが画面配信でき、視聴側は複数配信を同時に見られる `MUST`
  - 性能見積の基準は「4K 60fps × 同時 3 本」とする(architecture.md の試算参照)
- 視聴体験は「注視タイルは高画質、その他は simulcast で自動降格」を基本とする `MUST`
  - 全タイル常時最高画質はクライアントのデコード能力的に非現実的なため
- 画面共有の 4K 60fps 対応 `SHOULD`
  - Element Call / LiveKit の既定プリセットでは出ないため、クライアント fork での改修が前提
- 音声 Opus 384kbps(ステレオ)対応 `SHOULD`
  - 送信側クライアントの設定であるため、改修版クライアント統一(§7)が前提
- 通話の E2EE を標準にする `SHOULD`
  - メディアが借り物の VPS を通過するため(§6)
- ML ベースのノイズ抑制(Discord の Krisp 相当)`SHOULD`(2026-07-06 追加)
  - 方式は **RNNoise WASM に決定・実装済み**(同日。Krisp SDK は LiveKit Cloud 契約限定ライセンスのため除外)。既定 ON・トグル付き。残は運用者の聴感評価(roadmap Phase 8 参照)
- TURN の整備、UDP が通らないネットワークからの参加 `OUT`(当面)
- モバイルクライアント対応 `OUT`(当面)

## 4. ハイレゾ音声(別系統)

本節の系統は、本体(クライアント fork)とは結合しない**独立した拡張オプション**として提供する
(2026-07-05 決定、§9 決定記録参照)。クライアント fork への組み込み(起動導線・自動ミュート)は行わない。

- 主用途: 通話音声そのものの高音質化。会話が成立する片道遅延(目標 150ms 以内)を最優先とする `MUST`
- 192kHz / 24bit 対応 `SHOULD`
  - 音声帯域は可聴域を大きく超えるため知覚差は限定的だが、規格値として担保する方針(発案者判断として記録)
- WebRTC(Opus)系統との併用: ハイレゾ対応者同士はハイレゾ、非対応者へは Opus にフォールバックする `SHOULD`
  - 二重再生の防止が必要。クライアントへの自動ミュート実装は行わず、運用ルール(利用ガイドに明記)で代替する(拡張オプション化に伴う再スコープ、2026-07-05)
- 送信側は専用ネイティブアプリ + 対応オーディオインターフェース前提 `MUST`
  - ブラウザのマイク取得は実質 48kHz 止まりのため
- 参加者はヘッドセット必須(運用ルール) `MUST`
  - ネイティブ経路はブラウザのエコーキャンセルを通らないため。AEC の自前実装は `LATER`
- 伝送は可逆圧縮(FLAC 等)を既定とする `SHOULD`。非圧縮 PCM は帯域試算(architecture.md 参照)を満たす場合のみ `LATER`
- 実装は既製(SonoBus / JackTrip 等)のスパイクを先に行い、買う/作るを判断する(roadmap 参照)
- 映像とのリップシンク保証 `OUT`(当面)。音声主体の設計とする
- 384kHz / 32bit `LATER`

## 5. アカウント所有と連合

- Synapse + PostgreSQL でデータを自己保有する `MUST`
- 自分が管理するアカウントは自分のもののみ。他の参加者は Matrix federation で接続する `MUST`
- 友人は本スターターで自前サーバーを建てて自己管理するか、任意のホームサーバーでアカウントを作る `MUST`
- 自前サーバーを建てる友人は、SFU を含む MatrixRTC backend も自前で用意する `MUST`
  - 通話は「最初にセッションへ参加した人の SFU」でホストされる(MatrixRTC の focus 選択。各クライアントは自分のホームサーバーの `.well-known` から SFU を自動発見し、最古参加者の SFU が全体で採用される、いわゆる早い者勝ち方式)。互いの通話へは federated ユーザーとして参加できる
- 自前サーバーを持たない参加者(他ホームサーバーのアカウントのみ)も通話の開始(=最初の参加)自体は技術的に可能(2026-07-06 実装確認: room power level 等による制限は設けていない)。ただし先に参加すると相手のホームサーバーの SFU が使われてしまうため、自サーバーの SFU で運用したい場合は自ホームサーバーのアカウントを先に通話へ参加させる運用でカバーする `SHOULD`

## 6. ネットワーク・インフラ

- デプロイ形態は次の 2 つを一級サポートする `MUST`
  - A: 自宅サーバー + VPS(データは自宅、edge と SFU は VPS。以下の各項はこの形態の記述)
  - B: VPS 単独(全コンポーネントを 1 台の VPS に集約)
  - 形態 B ではデータが VPS 上に載るため、§1 のログ保全を満たす条件として VPS 外への定期バックアップを必須とする
- HTTP 系の入口: 外部 → Cloudflare → VPS → Tailscale → 自宅サーバー `MUST`
- メディア(UDP)は Cloudflare を迂回し、VPS で直接終端する。SFU(LiveKit)とハイレゾ音声サーバーは VPS に配置する `MUST`
  - Cloudflare の proxy は HTTP(S)/WebSocket のみで WebRTC の UDP を通せないため
  - Tailscale 区間には HTTP 系のみを通し、メディアを通さない
- VPS は国内リージョンを推奨。回線帯域は当面選定条件にしない(回線は将来変更予定。参考試算は architecture.md)
- VPS には永続データを置かない(ステートレス)。構築はスクリプト化し、消えても再構築できる状態を保つ `MUST`
- Cloudflare 経由の Synapse にはプランのアップロード上限(無料枠 100MB)が効く。`max_upload_size` を上限内に収めるか、メディアリポジトリのパスのみ Cloudflare を外す `MUST`
- 自宅回線は上下 10Gbps。ただし本構成ではメディアは自宅を通らない(帯域のボトルネックは VPS 側)

## 7. クライアント

- 参加者全員が改修版クライアントを使う。他クライアントの互換性は考慮しない `MUST`
  - Matrix 側に求めるのはアカウント(認証と federation)のみ
- 第一候補は Cinny fork。採用可否はスパイク(roadmap Phase 2a)の判断ゲートで確定する
- fork の改修対象:
  - Discord 風の UI / 文言 / テーマ(従来方針)
  - Element Call 層のメディアパラメータ(4K プリセット、Opus 384kbps)
  - ハイレゾ系統の起動導線は fork 改修対象から除外(拡張オプション化、2026-07-05。§9 決定記録参照)

## 8. 非機能

- 再現性: イメージタグ固定と Renovate 等による更新 PR 運用 `MUST`
- セットアップの編集ゼロ化(homeserver.yaml の自動パッチ) `MUST`
  - 友人が同じスターターを建てる前提のため、ドキュメント品質とセットアップ容易性は機能要件に準ずる
- CI: shellcheck / compose 検証 / Caddyfile 検証 `SHOULD`
- バックアップ: PostgreSQL dump、media、signing key、E2EE リカバリーキー、および復元訓練 `MUST`
- 可用性の単一障害点(VPS 等)は個人運用として許容する。代わりに復旧・再構築のスクリプト化を必須とする `MUST`

## 9. 決定記録(2026-07)

- **Discord パリティ方針 (2026-07-06)**: 目指すのは Discord 代替なので、**Discord にある機能は基本対応する**。個別機能の「対応しない」判断は例外として理由付きでここに記録する。この方針の第一適用としてノイズ抑制 (Krisp 相当) は「対応する」で確定し、RNNoise WASM で実装済み (§3)

- 「BAN 耐性」の意味を明確化: 守る対象はアクセス継続ではなくログの保全(チャット履歴・メディア・在籍記録)。アクセス喪失は許容する
- 通話・配信を要件の中心に据え、MatrixRTC(Element Call + LiveKit)標準構成を採用する
- ハイレゾ音声は WebRTC では原理的に不可(Opus は 48kHz 固定)のため別系統に切り出し、目標を 384kHz/32bit から 192kHz/24bit に引き下げた
- ハイレゾ系統の主用途は「通話音声そのものの高音質化」と確定(音楽鑑賞・楽器セッションではない)
- ハイレゾ音声は本体(クライアント fork)と結合せず、独立した拡張オプションとして提供する(2026-07-05)。fork への起動導線・自動ミュート制御は実装せず、**独立リポジトリ selfmatrix-hires**(JackTrip hub の provision スクリプト + 利用ガイド)として本スターターからも分離する(同日決定)。二重再生防止と人数上限は運用ルールで担保する
- SFU は自宅ではなく VPS に置く(メディアの公開終端が VPS になるため、トンネル往復を避ける)
- モバイル・TURN・リップシンクは当面対象外と明示した
- 通話からカメラ映像を外し、性能見積を「4K 60fps × 3 本」に更新
- VPS の回線帯域を当面の選定条件から外した(回線は将来変更予定)
- 可用性の単一障害点を明示的に許容し、復旧・再構築のスクリプト化を必須要件とした
- 自前サーバーを建てる友人は SFU も自前で用意する方針に変更(通話開始権限の未決定を解消)
- デプロイ形態として「自宅 + VPS」「VPS 単独」の 2 つを一級サポートする
