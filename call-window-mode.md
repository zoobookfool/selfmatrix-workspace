# 別ウィンドウ通話開始モード (設計検討、2026-07-07)

**ステータス: 検討ドラフト v0.1** — 運用者と AI (Claude) の検討結果。既定値 1 点が未確定 (§5)。
実装はレビュー/テストの区切り後 (担当未定: GPT or Claude)。確定したら ui-design-notes v1.5 に反映する。

## 1. 背景と目的

- 通話 UI 全体ポップアウト (実装済み) は「動いている通話 (widget iframe) を後から別窓へ移す」ため、
  hangup → 再 join (~1.4 秒) が**原理的に**避けられない (iframe の DOM 移動 = 強制リロード、
  popout-spike.md で実証。LiveKit は同一 identity の二重参加も許さない)
- **最初から別ウィンドウで join すれば移動が発生しないので、再接続はゼロ**。
  「通話 UI とチャット UI を同時に見て・触る」(v1.3 合意の目的) を最初から満たせる
- 実装は既存 CallPopout の流用 (hangup の儀式を省くだけ)。**cinny 側のみで完結、EC 変更不要**

## 2. 決定事項 (2026-07-07 運用者回答)

| 論点 | 決定 |
| --- | --- |
| 参加の導線 | **通常は単一の「参加」ボタン (案 B の見た目)**。設定で「参加時に毎回選ぶ (分割ボタン = 案 A)」表示に切替可能 |
| 既定の開き方 | 設定項目にする (このウィンドウ / 別ウィンドウ)。**製品としての初期値は運用者の好みに合わせる — §5 で要確定** |
| 設定の保存先 | **二層**: アカウント既定 (Matrix account data、端末間で引き継ぐ) + 端末オーバーライド (localStorage、その端末だけ違う設定にできる)。「PC の新規セットアップで設定し直したくない、でも端末ごとに変えたい」の両立 |
| 初回セットアップ | **含めない** (うるさくしない原則。設定画面で変更) |

## 3. 動作仕様案

### 参加フロー (別ウィンドウで開く場合)

1. 参加クリックの**同期スタック内**で `window.open` (ポップアップブロッカー対策、既存パターン。
   target は既存の `cinny_call_popout` を再利用)
2. 別窓に「接続中…」プレースホルダー → CallPopout として**初回 join** (再接続なし)
3. メインウィンドウは既存の「通話は別ウィンドウで進行中」バナー + 通常のチャット UI

### 設定 (cinny 設定画面に 2 項目)

- **通話の開き方**: このウィンドウ / 別ウィンドウ
- **参加時に毎回選ぶ**: OFF (既定) = 単一参加ボタンが上記既定に従う / ON = 参加ボタンが分割になり毎回選べる

保存は二層: 設定変更は account data (`in.selfmatrix.call_window_mode` 等の新イベントタイプ) に書き、
「この端末だけ別にする」トグルで localStorage オーバーライド。読み取りは localStorage → account data → 既定値 の順。

### 各操作の挙動

- **別窓を閉じる = 通話から退出** (現ポップアウトと同じ。「閉じたらメインに戻す」は再接続を強いるので採らない)
- **「このウィンドウに戻す」ボタンは残す** (メインのバナー内)。ただしこれは移動なので再 join (~1.4 秒) を伴う —
  ツールチップ等で「再接続します」と明示する
- **メインウィンドウを閉じる/リロード = 通話窓も道連れ** (Matrix クライアントがメインに居るための制約。現仕様どおり)
- **途中ポップアウト (⧉) は併存**: このウィンドウで参加 → あとから出す経路も従来どおり (こちらは再接続あり)
- **すべての参加経路が設定に従う**: Prescreen の参加ボタン、ルームナビの 2 クリック発信、着信応答
- **通話中の再参加/チャンネル移動**: target 名固定により同じ窓を再利用して次の通話を開く (窓が増殖しない)
- **ポップアップブロック時のフォールバック**: エラーで止めず**このウィンドウで通常参加**し、
  「ブロックされたためこのウィンドウで開きました」と通知 (通話成立を優先)

### 補助機能との関係

- **ストリーム単体 🗗 (再接続なし)**: 通話ウィンドウからさらに別窓へ出せる (窓 from 窓は user gesture があれば可)。
  **保留中の「グリッドタイルへの 🗗 追加」はこのモード確定後に再開する** (roadmap Phase 8 残課題)
- E2EE・権限は通常 join と完全に同一 (このモードは join する場所が違うだけ)

### nice-to-have (初期実装ではスコープ外でよい)

- 通話窓のサイズ/位置を端末ローカルに記憶し、次回同じ場所に開く (現状は 960x640 固定)
- 通話窓のタイトルにチャンネル名 (実装済みの `Call — <room>` を踏襲)

## 4. 技術メモ (実装者向け)

- 中核: `cinny/src/app/hooks/useCallEmbed.ts` の `useCallStart` に分岐を足す —
  開き方が「別ウィンドウ」なら `openCallPopoutWindow()` → `new CallPopout(...)` を直接生成
  (既存 `useCallPopout` から hangup 待ちを除いた形)。`CallPopout`/`PoppedOutBanner`/所有権ガードは実装済みのものを流用
- 設定読み書きは `screenShareSettings.ts` / `shellLayout.ts` のパターンを合成
  (localStorage 契約 + account data + `makeShellLayoutContent` 型のマージ)
- 分割ボタン UI は Prescreen の参加ボタン部。folds の既存部品で
- テスト: 設定の二層解決 (override 優先) の unit + E2E (別窓で join → メインにバナー、再接続が無いこと =
  join が 1 回であること)

## 5. 未確定事項
