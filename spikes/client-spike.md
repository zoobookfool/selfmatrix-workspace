# Client spike(roadmap Phase 2a)

目的: Cinny fork の採用可否を、改修着手前に実測で判断します。タイムボックスは 5 営業日。延長するより「不合格として Element Web fork の比較に進む」判断を優先します。

## 検証環境

実デプロイ(Phase 1 / Phase 3)を待たず、ローカルで完結させます。

- MatrixRTC backend: element-call リポジトリ同梱の dev backend(docker compose)を使う
  - Synapse が 2 台(`synapse.m.localhost` と `synapse.othersite.m.localhost`)含まれるため、federated 参加の検証(項目 3)もローカルで行える
  - ローカル CA 証明書をブラウザに信頼させる手順が element-call の README にある
  - リポジトリに公開されたテスト用シークレットを使う構成のため、ローカル限定で使い、インターネットに公開しない
- クライアント: cinny を upstream のまま clone し、README に従い dev サーバーで起動する。`config.json` の homeserver を dev backend に向ける
- 参加者のシミュレーション: ブラウザプロファイル 3〜4 個(または別端末)でユーザーを分ける

## 検証項目

### 1. 複数人同時画面共有

- 手順: ビデオルームに 3 ユーザーで参加し、全員が画面共有を開始する
- 合格: 3 本の共有が同時に表示され、任意のタイルを注視(拡大)できる
- 不合格の例: 共有が 1 本に排他される、他人の共有開始で自分の共有が止まる

### 2. メディアパラメータの上書き到達性(最重要)

- 手順: Cinny に埋め込まれた Element Call / livekit-client 層で、画面共有の解像度・フレームレートと音声ビットレートを決めているコードを特定する。値を変えたローカルビルドを作り、`chrome://webrtc-internals` で送信パラメータの変化を実測する
- 合格: 差し込み点が特定でき、変更がビルドと実測の両方に反映される。成果物として該当ファイルパスと差し込み方法をメモに残す
- 不合格の例: パラメータが依存パッケージや iframe の奥に固定されており、fork 側から現実的に触れない

### 3. federated アカウントの通話参加

- 手順: `othersite` 側 Synapse のユーザーで、こちら側 Synapse のユーザーが開始した通話ルームに参加する
- 合格: 音声と画面共有が双方向に成立する
- あわせて確認: SFU を持たない federated ユーザーが通話を「開始」できない挙動が、requirements.md §5 の想定どおりか
  (【追記 2026-07-06】この想定はのちに訂正 — 実際は「最初のセッション参加者の SFU が使われる早い者勝ち」で、federated ユーザーも技術的には最初の参加者になれる。requirements.md §5 の現行記述を正とする)

### 4. ボイスチャンネル運用の UX

- 手順: 常設ビデオルームを「ボイスチャンネル」に見立て、入退室・再入室・ミュート・共有の切り替えを一通り行う
- 合格: Matrix 用語を知らない参加者に説明できる程度に操作が素直であること(主観評価で可)。DM 通話非対応は許容済み

## 記録と判定

- 各項目について「結果 / 根拠(スクリーンショットやファイルパス) / 気づき」をメモに残す(ローカルメモで可)
- 4 項目合格 → Cinny fork 続行(Phase 2b へ)
- 項目 2 が不合格 → Element Web fork で同項目を検証し、比較して判断する
- 項目 1 / 3 / 4 の不合格 → 内容次第。upstream の issue / PR の状況を確認し、待つ・自分で直す・乗り換えるを判断する
