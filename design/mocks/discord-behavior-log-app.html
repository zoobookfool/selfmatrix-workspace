Discord 動作ログアプリ実測版

SelfMatrix「UI 総点検 (Discord 実物突き合わせ)」の基準器。 discord-behavior-log.html (web 実測版、2026-07-13) の姉妹編で、 Discord Windows デスクトップアプリでどう見えて・どう動くかを観察した記録。 デザイン提案ではない。web 版と重複する項目は差分のみ、web 版に無い項目 (通話ビューのホバー chrome など) は新規に記録する。

方法論

  • 実測日: 2026-07-14、Discord Windows デスクトップアプリを対象に観察
  • キャプチャ手法: PrintWindow によるウィンドウキャプチャ (DOM 越しの DevTools 実測ではない)。 表示スケール 150% 環境のため、物理px→CSS px は ÷1.5 で換算
  • 観察のみ: 非公開 API・DOM 改変・裏技的操作は行っていない。見えている挙動をそのまま記録
  • 秘匿: 実サーバー名・実ユーザー表示名・ゲームタイトルは非掲載。再現図の名前は 「User A」等のプレースホルダ、自分自身は「運用者」と表記。チャンネル名もプレースホルダ (Voice Lounge はローカル開発環境の部屋名で、観察対象の実サーバーとは無関係)
  • アセット非使用: Discord のロゴ・アイコン画像・音は一切含まない。CSS・図形・絵文字のみで近似再現
  • web 版との対応: 同一トークン・寸法は「一致」、値が異なる/web 未測定の項目は各節の obs-note で明示する
  • 情報源の種別: 本文中の記述は4種類を区別する — 「実測」(画像からの直接観測)、「証言」 (Discord実ユーザーである運用者本人からの申告)、「推測」(未確認の解釈。断定はしない)、 「未確認」(裏取りできていない事項)。証言・推測・未確認はいずれも実測ではないため、その旨を 明記した上で参考情報として扱う

web版との差分・未確認点

音量調整の置き場所は web/app 共通: 参加者右クリックメニュー内のスライダーが本体 (web実測版§5で実測済み)。映像上にスライダーが浮くという 想定は誤りだったため、ここでは差分として扱わない。アプリ版の右クリックメニュー自体は§5で宿題として 別途確認する。
タイル単位ミュートボタン (🔇) はアプリのホバーで確認できたが、web に同じものがあるかは 今回未確認 (web実測版のホバーデモは通常メッセージ行が対象で、通話タイルのホバーは検証していない)。
画質ピル「720p 30 FPS」がヘッダーに常設表示される (ホバー時)。web実測ログには対応する 記録が無い (web側未確認 — 見落としの可能性と、そもそも表示されない可能性の両方が残る)。
ポップアウト (⧉) はコントロールバー右端 (全画面の隣) にあり、ステージ上ではなくビュー全体の chrome に属する。web実測版§7では「配信者側の操作範囲では document PiP は一度も開かなかった」までが実測だった。アプリ版では§4で 「別OSウィンドウの全体ポップアウト」と「アプリ内PiPミニプレイヤー」の2メカニズムが実測できたが、 このアイコンがどちらを起動するのかは未確定のまま (§4 の todo-note を参照)。

再現度への留保

フィルムストリップの実キャプチャでは、ライブ (配信中) タイル3枚のうち2枚のみにタイル単位の ミュートボタン (🔇) が写っており、残り1枚には出ていなかった。全タイル共通の挙動か、特定条件下 でのみ出るのか、出現条件は未確認。
ステージ右下の小アイコン2〜3個 (スクリーンショット風カメラ・丸型・拡大鏡) は用途を明確に特定できて いない。上のデモでは形状のみ再現し、機能名の当て推量はしていない。

差分 (プレースホルダ露出、推測込み): タイムアウト/追放の表記が 「???(開発中) をタイムアウト」「???(開発中)を追放」— 本来ユーザー名が入るべき位置に 未置換のプレースホルダ文字列がそのまま露出していた。ローカライズ漏れ、または実験的機能のラベル漏れと 推測 (断定不可)。web実測時はこの2項目とも「タイムアウト」「追放」という単純な表記だった。

§6 テキストチャンネルの通話イベント有無

証言 + 実測
運用者証言 (2026-07-14): サーバー VC の出入り・通話開始/終了は、テキストチャンネルのタイムラインに 一切表示されない。「通話イベント行」という概念自体が Discord には存在しない、とのこと。

実測での裏取り (2026-07-14、証言と一致)

同一サーバーの VC で 16 時間超の通話が継続 (サイドバーの経過時間カウンタが 15:33:42 → 16:22:48 とキャプチャ間でカウントアップ、 その間に複数人の出入り・配信あり) した状況で、対応するテキストチャンネルのタイムラインを確認したところ 通話系イベント行はゼロだった。
同じタイムラインに「〇〇 がメッセージをチャンネルにピン留めしました。」というシステムメッセージ行は 表示されている — つまりシステム行という仕組み自体は Discord に存在するが、通話イベントには 一切使われないことが実測で確定した (証言→実測に格上げ)。
テキストチャンネル表示中の右側メンバーリストは、ロール別グルーピング+オフラインという構成。 通話中のメンバーは名前の下に「🔊 会話中」と表示される — 通話の状態はタイムラインではなく、 プレゼンス表示 (サイドバー/メンバーリスト) 側で見せるという Discord の設計方針を裏付ける。

証言と実測が一致したため、この項目の裏取りはここで完了とする。

§7 DMの通話表示 (存在せず — 証言)

運用者証言 (2026-07-14): DM でも「〇〇が通話を開始しました」等の通話メッセージはタイムラインに 表示されない。サーバー VC (§6) と同様、DM 側にも通話イベントのシステムメッセージという概念自体が無い、 とのこと。
正直な注記: 当初「DM/グループDM の通話でタイムラインにシステムメッセージが残る」という想定を 持っていたが、これは実測ではなく AI の学習知識ベースによる思い込みだった。今回この想定は検証できておらず、 旧仕様だった可能性・そもそもの誤認の可能性の両方が残る。少なくとも運用者の使用範囲では一度も観察された ことがない、という証言のみが手元にある。
裏取りは任意 — スクリーンキャプチャによる確認ができれば添えたいが必須ではない。今回のキャプチャ 対象がサーバー VC のみだったため、DM 画面そのものの実測は依然として未取得。
