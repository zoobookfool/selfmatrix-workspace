## エグゼクティブサマリー: Sonnet レビューからの主要な訂正

1. **【最重要・訂正】`matrixRTCMode` は「fork 改造なしのタダの解決策」ではない。** Sonnet は「設定 1 行変更で SFU 早い者勝ちが緩和される」としたが、2 つのレンズが独立に読んだ結果、機構は存在するものの**そのままでは通話が繋がらなくなる恐れがある**:
   - `multi_sfu` (Compatibility/Matrix_2_0) は各クライアントを**自分のホームサーバーの SFU** に繋ぐが、異なる SFU 間で音声・映像を中継するには LiveKit サーバー側の**ノード間リレー (Redis 共有の region 設定)** が必要。現状の dev バックエンドの 2 つの LiveKit はこの連携が無く、**別々のルームに分断されて相互に音が届かない可能性が高い** (帯域が増えるのではなく、繋がらない)。
   - `matrixRTCMode` は各ブラウザの localStorage の開発者向け設定で、**URL パラメータや cinny 側から一括注入できない** (全員が手動で Developer Mode を有効化して選ぶ必要がある)。
   - 一方でアーキ観点のレンズは「upstream 文言で要件は明記済み (相手 EC クライアントが v0.17.0 以上)、各クライアントは自分の transport を独立に advertise するだけなので**こちら側だけの設定で足りる可能性**もあり、1 行の低リスク改修」と評価。**2 つのレンズの結論が割れている** — これ自体が「机上では確定できない、実測が必須」というシグナル。
   - **結論**: Bot 常駐案より先に試す価値はあるが、Sonnet の「クリーンな勝ち」という性質付けは楽観的すぎる。**dev の 2 ホームサーバー構成 (synapse.m.localhost + othersite) で `multi_sfu` に切り替えて、異なるホームサーバー同士が本当に音声・映像を共有できるかを実測してから**判断すること。繋がらなければ LiveKit のリレー構築が必要になり「1 行」では済まない。

2. **【訂正】ノイズキャンセルの「設計欠陥なし」は評価軸が違う。** Sonnet は「echoCancellation/noiseSuppression 既定 true で欠陥なし」としたが、これはブラウザ標準の軽量ノイズ抑制の有無を答えているだけ。コードを検索した結果 **Krisp/RNNoise 相当の ML ベース音声処理は皆無** (`TrackProcessorContext` は映像の背景ぼかし専用)。ユーザーが「取り入れないとまずい」と明言した Discord Krisp との体験差は埋まっていない。要件文書に「対応する/しない」の記載自体が無いので、明示的な決定を残すべき。

3. **【新規・critical】プロダクトの中核価値がスケールで未実証。** roadmap は Phase 3/5 を「完了」表記だが、exit criteria の**「10 人 + 画面共有 3 本」「4K60×3 本 + 10 人」の負荷/品質試験は未実施**。「Discord への課金なしで高画質・高音質」という価値提案そのものが、実際に友人を集めるまで実証されていない。他の polish より優先度が高い可能性がある。

## 確定バグ (複数レンズで独立確認、優先度: 高)

### B1. 【新規】通知ゲートがルーム個別の「すべてのメッセージ」設定を無視して抑制する

Sonnet の 2 件 (#4) に加えて Opus が発見した**第 3 の通知バグ**。`ClientNonUIFeatures.tsx` の Discord 風ノイズ抑制ゲートは `NotificationType.Mute` しか早期 return せず、cinny に既存の per-room `NotificationType.AllMessages` (ユーザーが明示的に「この部屋は全メッセージ通知」に設定した状態) を一切見ない。結果、ユーザーが特定チャンネルを「すべて通知」にしていても、その部屋の通常メッセージは黙って抑制される。
- 修正: gate 条件に `getNotificationType(...) === AllMessages` の除外を追加 (`ClientNonUIFeatures.tsx:195-224`)。確度 high。

### B2. 【確認】通知ゲートの highlight 差分判定が誤爆/抑制する (Sonnet #4)

`unreadCacheRef` が `roomToUnreadAtom`/`RoomEvent.Receipt` と非同期なため、(a) 起動直後に未読メンションのある部屋へ通常メッセージが来ると誤って鳴り、(b) 他デバイスで既読後は本物のメンションが抑制される。**2 レンズが独立にコードトレースで再現確認** (確度 high)。Sonnet の原因分析・修正案 (cacheをroomToUnreadAtomでプレフィル + Receipt監視) は正確。

### B3. 【確認】shellLayout の連続変更で片方の設定が消える (Sonnet #5)

`General.tsx` が `setAccountData` を await せず、`makeShellLayoutContent` が `getAccountData` (sync エコー待ちで即時更新されない) からベースを読むため、サイドバー位置とナビ位置を素早く連続変更すると 2 回目が 1 回目を上書き消失。**2 レンズが独立確認** (確度 high)。修正案 (shellLayoutAtom の現在値をベースにする or 直列化) は妥当。

## 新規の高優先度所見 (Sonnet 未指摘)

### H1. element-call fork (v0.20.1 固定) が upstream の**セキュリティ修正込みで 147 コミット遅れている**

CLAUDE.md は v0.20.1 固定を「cinny の依存に合わせるため」とだけ説明するが、fork の分岐点 (v0.20.1-rc.1、2026-06-02) から upstream `livekit` HEAD (v0.21.0-rc.1、2026-06-29) までに 147 コミットの差があり、その中に **`Update dependency vite to v8.0.16 [SECURITY]`** を含むセキュリティ修正が含まれる。cinny 側は upstream に追従できている (merge-base が HEAD 一致) のに、**EC 側だけが「意図的固定」の名の下でパッチ不在期間を積み上げている**。
- 推奨: 「無期限固定」ではなく「セキュリティ修正をチェリーピックする棚卸し対象」として運用ルール化。四半期に一度 `git log --grep=SECURITY` で確認するタスクを operations.md に明記。確度 high。

### H2. 1 人運用での fork 持続可能性リスク: EC の変更速度が cinny の 4.7 倍で、しかも改修予定の中核ファイルに集中

直近 3 か月の upstream コミット数: cinny 122 件 vs **EC 575 件 (4.7 倍)**。EC で最も変更頻度が高いのは `InCallView.tsx` (38 回) と `CallViewModel.ts` (33 回) で、これはまさに design-mismatch ①②③ (ポップアウト操作 UI・`showNameTags$`・`naturalGridMode$`) の改修対象そのもの。
- 推奨: EC 側の fork 差分は**「別コンポーネント追加」方式を徹底**し、`CallViewModel.ts`/`InCallView.tsx` への直接パッチは薄いラッパー層で吸収。限られたメンテ時間は cinny より EC に重点配分。確度 high。

### H3. cinny fork の独自機能にテストが 1 つも無い (壊れても検知できない)

cinny には test スクリプトも `*.test.*` ファイルも存在しない (0 件。EC は 70 件以上)。B1〜B3 のような fork 独自ロジックのバグは**退行しても CI で捕まらない**。
- 推奨: 最低限 `shellLayoutAtom` のマージと通知ゲートの highlight 判定に vitest を導入 (cinny には vitest 自体の導入から)。次善策は手動 QA チェックリストの明文化。確度 high。

### H4. systemd 自動バックアップが **NOPASSWD sudo 前提を一切文書化しておらず、無言で毎晩失敗しうる**

`selfmatrix-backup.timer` は非 root で `backup.sh` を実行するが、中身は署名鍵/メディア読み取りに `sudo` を使う。systemd には TTY が無いため、対象ユーザーに NOPASSWD sudoers を事前設定しない限り sudo は毎回失敗する。README/operations/unit のどこにもこの前提の記載が無い (`grep` で NOPASSWD/sudoers/visudo は 0 件)。**「バックアップは動いている」と思い込んだまま実際は毎晩失敗、というシナリオが現実的**。
- 推奨: 必要な sudoers 行の例を README/operations に明記。`OnFailure=` 通知ユニット追加、または「定期的に `systemctl status` で確認」の明記。確度 high。

### H5. i18n の高頻度導線が広範に未 i18n (Sonnet のリストは所在が一部不正確 + 過小)

- **EC**: en 226 キー中 **120 キー (53%) が ja 欠落**を実測 (`error.*`/`lobby.*`/`group_call_loader.*`/`settings.devices.*`/通話中の参加人数・再接続中表示等)。`fallbackLng:"en"` のため生キー露出ではなく英日混在表示になる (Sonnet 未言及の緩和材料)。前回コミット 015c0633 の「upstream は ja 完備」という前提が誤りだった。
- **cinny**: en/ja のキーは完全一致 (0 件欠落) — cinny の問題は「キー欠落」ではなく**「`useTranslation` を使わない生英語ハードコード」**。新規発見の高頻度箇所: `FallbackContent.tsx` (「削除されました」「復号できません」「(edited)」= 日常的に表示)、ファイル添付/アップロード UI 一式、UIA 全ステージ (登録トークンエラー・パスワード再認証)。
- **優先度 (踏む頻度順)**: 登録トークンエラー画面 (下記 M4) → EC の `error.*`/`group_call_loader.*` → cinny `FallbackContent` → ファイル添付 → UIA → デバイス検証フロー → 絵文字パック編集。`DeviceVerificationSetup` 単体より上記が上位。確度 high。

## 中優先度の所見

### M1. 【訂正】ライトテーマの selfmatrix ブランドが存在しない — Sonnet の「lightThemeId を設定」は実行不可能

Sonnet #6 の対応案「`initialSettings` に `lightThemeId` を設定」は、**設定先となるライト版 selfmatrix テーマがそもそもコードに無い** (`useTheme.ts` の `SelfMatrixTheme` は Dark 固定)。実際にはライト版テーマの新規デザイン・実装が必要 (S ではなく M〜L 工数)。当面はシステムテーマ追従をオフ固定にする運用回避も選択肢。バグ自体 (システムテーマ ON + OS ライトで標準 LightTheme にフォールバックしブランド喪失) は確認済み。

### M2. E2EE リストア訓練が行数一致のみ検証、鍵復元可能性は未検証

`restore.sh --drill` は users/rooms/events の `count(*)` だけを見て PASS 判定。SSSS 暗号化鍵バックアップ・cross-signing 鍵がリストア後に整合し、クライアントが実際に復号できるかは範囲外。メモリの「Phase 4 完了、リストア訓練 PASS」は**行数一致という限定的な意味**で、E2EE 復旧を保証しない。
- 推奨: 「テストユーザーで E2EE メッセージ送信 → backup → restore → 復号できるか」の実地確認を最低 1 回追加。確度 high。

### M3. 登録トークンエラー画面が生 errcode を表示 — Phase 7 目玉機能の第一印象

`RegistrationTokenStage.tsx` が `title={errorCode}` でサーバー生エラーコード (M_UNKNOWN 等) をそのままダイアログタイトルに出し、無効/期限切れ/使用済みを区別できず英語。**「招待した友人が最初に触る画面」**(Discord 移行しやすさの核) の失敗時体験として、i18n の中でも優先度を上げるべき (Sonnet は ux 分類だが reprioritize)。

### M4. `backup.sh` が署名鍵ファイル複数存在時に tar ごと失敗

`$(cd synapse/data && sudo ls ./*.signing.key)` は `*.signing.key` が 2 つ以上あると改行を畳んで単一引数化し tar が stat 失敗 (実際にサンドボックスで再現、exit 2 で `set -euo pipefail` により全体停止)。通常は鍵 1 つで発生しないが、鍵ローテーションや旧鍵残置で踏む。フェイルセーフ (サイレント破損でなく即エラー) だが未文書。

### M5. 画質/FPS の「落とす」安全弁の実地確認 (低スペック参加者対策)

画質/FPS 選択 UI が無く常に 4K60/384kbps 固定 (コード確認済み)。課金壁なしの高画質固定は**意図的な差別化**として現状維持でよいが、低スペックの友人が参加した際に adaptiveStream/simulcast の自動降格が実際に効くかは未検証。「高画質を謳ったのに固まる」を避けるため、**UI をフル実装するより「重い時に自動で落ちる安全弁があるか」を負荷試験 (C1) と併せて確認**する方が費用対効果が高い。

### M6. VPS 復旧手順の定期再訓練スケジュールが未設定

単一障害点 (VPS) の許容は明示的合意済みで妥当だが、`restore.sh --drill` は「1 回実証」であり、依存更新や homeserver.yaml スキーマ変更で陳腐化しうる。月次/四半期の定期タスク化を operations.md に明記推奨 (確度 low、探索範囲外の手元 cron は未確認)。

## Opus の推奨アクション順 (判断は運用者に委ねる)

1. **負荷/品質試験 (C1)** — プロダクトの価値提案を実証する最優先ゲート。友人が実際に同時通話する最初の機会を試験本番に充てる手も。これで M5 (画質降格の安全弁) と VPS 増強要否も同時に判断できる。
2. **`matrixRTCMode` の federation 実測 (C2)** — Bot 常駐案の前に、dev 2 ホームサーバー構成で `multi_sfu` が本当に繋がるか実測。繋がるなら安価な勝ち、繋がらないなら Bot 常駐 or リレー構築へ。**「繋がらない」可能性を織り込んで検証する**こと。
3. **確定バグ B1〜B3** — 通知ゲート 3 件は「動作が明確に間違い」。B1 (AllMessages 無視) と B2 は同じファイルなので一括。B3 は別ファイルだが小さい。
4. **セキュリティ衛生 H1 (EC のセキュリティパッチ棚卸し) + H4 (バックアップ sudo 前提の文書化)** — 放置すると気づかないまま実害が蓄積する類。
5. **i18n H5** — 踏む頻度順に。登録トークンエラー (M3) は Phase 7 目玉の第一印象なので最初に。
6. **fork 持続性 H2 + テスト H3** — design-mismatch ①②③ に着手する前に「別コンポーネント方式」を決め、同時にテスト土台を入れると退行を防げる。
7. **ノイズキャンセル (M2)** — ユーザー重視。要件文書に「対応する/しない」の決定を残し、やるなら LiveKit Krisp SDK / RNNoise WASM のスパイクを roadmap に。
8. 残りの M/L は上記に着手するついでにまとめて。
