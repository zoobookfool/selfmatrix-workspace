## Findings

### [P1] Route A generated nginx proxies Synapse to the wrong backend port

In the Route A home-side override, Synapse is exposed as `${BACKEND_BIND_IP}:8028 -> 8008`.

Evidence:

- `docker-compose.route-a.example.yml` says `synapse` is exposed on `8028`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/docker-compose.route-a.example.yml#L15-L18
- Actual port mapping is `8028:8008`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/docker-compose.route-a.example.yml#L27-L29
- But the VPS edge nginx generated by `scripts/provision-rtc-vps.sh --with-edge` proxies Matrix traffic to `${HOME_BACKEND_IP}:8008`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/provision-rtc-vps.sh#L248-L257

Dry-run confirmation:

```text
server_name matrix.example.com;
location / {
    proxy_pass http://100.64.0.1:8008;
}
```

Impact: following the documented Route A provisioning path leaves `matrix.example.com` pointing at a port that is not published by the home-side override, so Matrix client/federation API traffic fails.

Suggested fix:

```nginx
proxy_pass http://${HOME_BACKEND_IP}:8028;
```

Also consider defining a single variable for the home Synapse backend port so docs, compose, and generated nginx cannot drift again.

### [P1] Generated nginx edge does not block `/_synapse/admin`

The repo's Caddyfile blocks Synapse Admin API at the public edge, and `docs/operations.md` states that public edge reverse proxies should deny `/_synapse/admin`. However, the nginx config generated by `scripts/provision-rtc-vps.sh --with-edge` uses a catch-all `location /` for the Matrix host and does not include any deny rule for `/_synapse/admin`.

Evidence:

- Caddy blocks admin API:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/caddy/Caddyfile#L23-L25
- Operations doc requires edge blocking:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/docs/operations.md#L113-L127
- Generated nginx only has catch-all Matrix proxy:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/provision-rtc-vps.sh#L248-L257

Impact: users who follow the Route A nginx provisioning path can unintentionally expose Synapse Admin API routes to the Internet. The API still needs admin credentials, but exposing this path contradicts the repository's own security model and increases attack surface.

Suggested fix: add this before `location /` in the generated `MATRIX_HOST` server block.

```nginx
location ^~ /_synapse/admin {
    return 403;
}
```

### [P1] Generated nginx creates the well-known directory but does not serve Matrix well-known JSON

In Route A, the home-side override disables the local Caddy service, so the VPS edge must serve `/.well-known/matrix/server` and `/.well-known/matrix/client`. The Caddyfile does this with inline JSON responses. The generated nginx config only sets `root /var/www/selfmatrix-well-known;` and creates the directory, but it never writes the required JSON files.

Evidence:

- Caddy serves required well-known JSON:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/caddy/Caddyfile#L6-L10
- Generated nginx only points to a root directory:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/provision-rtc-vps.sh#L237-L245
- The script only creates the directory:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/provision-rtc-vps.sh#L300

Impact: Matrix server/client discovery fails for Route A deployments created by the provisioning script unless the operator manually adds files that the script does not mention creating.

Suggested fix: either generate the files:

```json
{"m.server":"matrix.example.com:443"}
```

```json
{"m.homeserver":{"base_url":"https://matrix.example.com"}}
```

or use nginx `return 200` blocks with `default_type application/json` and `Access-Control-Allow-Origin *`.

If RTC is enabled, the generated client well-known should also optionally include `org.matrix.msc4143.rtc_foci`.

### [P2] `.env` is sourced as shell code and then written into YAML by string interpolation

Several scripts read `.env` using `source`. That means values are interpreted by the shell, not parsed as data. A password containing shell metacharacters, command substitution, quotes, or newlines can break the scripts, mutate values, or in the worst case execute shell code if `.env` is not fully trusted.

Evidence:

- `generate-synapse-config.sh` sources `.env`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/generate-synapse-config.sh#L17-L22
- `backup.sh` sources `.env`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/backup.sh#L14-L17
- `restore.sh` sources `.env`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/restore.sh#L28-L31
- The PostgreSQL password is then inserted into YAML with manual string formatting:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/generate-synapse-config.sh#L63-L71

Impact: a "long random password" may include characters that are legal in secrets but unsafe under shell/YAML interpolation. This can cause generated `homeserver.yaml` to be invalid or contain a password different from the Docker Compose database password.

Suggested fix:

- Parse `.env` as key/value data rather than sourcing it. Existing scripts such as `generate-cinny-config.sh` already avoid `source` for this reason.
- Validate domain and DB identifier variables before using them in SQL/YAML.
- Use a YAML library/emitter for `homeserver.yaml` edits instead of building YAML with f-strings.

### [P2] Backup artifacts containing secrets rely on default umask and are not encrypted for off-host sync

`scripts/backup.sh` writes DB dumps, `homeserver.yaml`, signing keys, and media archives under `backups/<timestamp>`. It later `chown`s the backup set to the operator, but it does not set a restrictive `umask`, create the directory with mode `0700`, or `chmod` the resulting files. With a typical `022` umask, some output files can become group/world-readable on the host.

Evidence:

- Backup directory is created without explicit mode:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/backup.sh#L19-L22
- Secret-containing config/signing-key archive is created:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/backup.sh#L29-L37
- Ownership is changed, but no permissions are tightened:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/scripts/backup.sh#L46-L50
- Off-host sync is documented as plain `rclone sync backups/ ...`:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/docs/operations.md#L151-L163

Impact: local users or backup storage operators may be able to read DB contents, homeserver config, and signing keys. Losing the signing key is already called out as catastrophic; exposing it is also serious.

Suggested fix:

```sh
umask 077
install -d -m 700 "$OUT"
```

After creating archives, enforce:

```sh
chmod -R go-rwx "$OUT"
```

For off-host backups, recommend encrypted storage such as `rclone crypt`, restic, borg, age-encrypted archives, or equivalent.

### [P2] `.env.example` overrides the pinned Synapse fallback with `latest`

`compose.yaml` has a safer fallback of `SYNAPSE_TAG:-v1.155.0`, but the documented quick-start flow copies `.env.example`, and `.env.example` sets `SYNAPSE_TAG=latest`. That means ordinary deployments use `latest`, not the pinned fallback.

Evidence:

- `.env.example` sets latest:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/.env.example#L11-L14
- `compose.yaml` fallback is pinned for Synapse:  
  https://github.com/zoobookfool/selfmatrix/blob/07679d5dc0a944bea1d945cc36976b5c05121e0b/compose.yaml#L19-L30

Impact: Synapse upgrades can include database migrations and operational changes. Using `latest` by default makes deploys and restarts less reproducible and can surprise operators after image pulls.

Suggested fix: set `SYNAPSE_TAG=v1.155.0` in `.env.example` or remove the variable from `.env.example` so the Compose fallback applies. Let Renovate open explicit version bump PRs.
