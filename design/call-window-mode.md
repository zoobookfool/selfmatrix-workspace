## 1. 前提知識 (このプロジェクトを知らない読者向け)

- **SelfMatrix** は Matrix プロトコル上に Discord 代替のチャット + 通話体験を作るプロジェクト。
  クライアントは 2 つのフォークの組み合わせ:
  - **cinny フォーク** = シェル (Discord 風のサーバー/チャンネル UI、チャット、通話コントロールバー)
  - **Element Call (EC) フォーク** = 通話画面本体 (タイル・配信表示)。**cinny のページ内に iframe
    (Matrix widget) として埋め込まれる**。通話の WebRTC 接続 (LiveKit SFU への接続) はこの iframe の中で生きている
- **ブラウザの制約 (この設計の出発点)**: iframe を別ウィンドウへ DOM 移動すると**必ずリロードされる**
  (HTML 仕様)。つまり「動いている通話画面を、接続を保ったまま別ウィンドウへ引っ越す」ことは不可能。
  また LiveKit は同一ユーザーの二重参加を検知すると古い接続を切る
- **実装済みの関連機能 2 つ** (混同注意):
  1. **通話 UI 全体ポップアウト**: 通話中に ⧉ ボタンで通話画面を丸ごと別ウィンドウへ「移す」。
     上記制約のため実体は「退出 → 別ウィンドウで作り直して再参加」で、**~1.4 秒の再接続を伴う**。
     ポップアウト中、メインウィンドウは「通話は別ウィンドウで進行中」バナー + 通常のチャット画面になり、
     「このウィンドウに戻す」ボタン (これも再接続) がある。別ウィンドウを閉じる = 通話から退出
  2. **配信ストリーム単体ポップアウト 🗗**: web/ECでは1つの配信映像を明示操作で別ウィンドウに出し、
     受信済みトラックを追加のvideo要素へattachするため再接続しない。native host連携は未実装
- 運用者の要望: 「通話画面とチャットを同時に見て・触りたい」「再接続は発生させたくない」

## 2. 提案: 最初から別ウィンドウで参加する

「後から移す」から再接続が生まれるのだから、**参加した瞬間から通話を別ウィンドウで開けばよい**。
初回参加がそのまま別ウィンドウ内で行われるだけなので、移動が存在せず**再接続はゼロ**。
メインウィンドウは参加直後からチャット画面のまま使える。

### 再接続の発生条件 (重要な整理)

| 操作 | 再接続 |
| --- | --- |
| **参加時に最初から別ウィンドウで開く (本提案)** | **ゼロ** — 初回参加がそのまま別窓で行われるだけ |
| 参加後に途中で別窓へ出す (既存の ⧉ ポップアウト) | あり (~1.4 秒) — 動いている iframe は移動できない |
| 別窓からメインへ戻す (「このウィンドウに戻す」) | あり (~1.4 秒) — 同上 |
| 別窓を閉じる | **なし** — 「閉じる = 通話から退出」(終了であって移動ではない) |

参加導線の UI の形 (後述の案 A/B/C) は「開き方を選ぶ方法」の違いであり、再接続の有無を左右しない。
再接続を払うのは「途中で移動する」操作のみ。既定を別ウィンドウにすれば移動自体が不要になる。

## 3. 決定事項 (2026-07-07 運用者確定)

検討した導線案: 案 A = 参加ボタンを分割して毎回選ぶ / 案 B = 設定のみ (参加ボタンは 1 つ) /
案 C = 設定 + 毎回選べる小導線。

| 論点 | 決定 |
| --- | --- |
| 参加の導線 | **通常は単一の「参加」ボタン (案 B の見た目)**。設定で「参加時に毎回選ぶ (分割ボタン = 案 A)」表示に切替可能 |
| 既定の開き方 | 設定項目 (このウィンドウ / 別ウィンドウ)。**初期値は「別ウィンドウ」で確定** — チャット同時利用 + 再接続ゼロが動機。Discord の挙動 (同一ウィンドウ内) とは意図的に異なる差別化なので、実装時に要件文書のパリティ方針の例外として記録する |
| 設定の保存先 | **二層**: アカウント既定 (Matrix の account data = サーバー保存でどの端末にも引き継がれる) + 端末オーバーライド (ブラウザの localStorage = その端末だけ違う設定にできる)。「PC の新規セットアップで設定し直したくない、でも端末ごとに変えたい」の両立 |
| 初回セットアップ案内 | この設定は**含めない** (初回案内をうるさくしない方針。設定画面で変更) |

## 4. 動作仕様

### 参加フロー (別ウィンドウで開く場合)

1. 参加クリックの**同期処理内**で `window.open` — ブラウザのポップアップブロッカーは
   ユーザー操作の同期スタック内の open しか許可しないため (実装済みポップアウトと同じ手法。
   ウィンドウ名も同じものを再利用し、窓が増殖しないようにする)
2. 別窓に「接続中…」プレースホルダー → その窓の中で通話 widget (iframe) を生成して初回参加
3. メインウィンドウは参加直後から「通話は別ウィンドウで進行中」バナー + 通常のチャット UI
   (実装済みポップアウトの状態管理をそのまま流用)

### 設定 (シェルの設定画面に 2 項目)

- **通話の開き方**: このウィンドウ / 別ウィンドウ (初期値: 別ウィンドウ)
- **参加時に毎回選ぶ**: OFF (初期値) = 単一参加ボタンが上記既定に従う / ON = 参加ボタンが分割になり毎回選べる

読み取り優先順: 端末オーバーライド (localStorage) → アカウント既定 (account data) → 初期値。
設定画面での変更はアカウント既定へ書き、「この端末だけ別にする」トグルで端末オーバーライドを書く。

想定スキーマ (実装時に型名・保存関数へ落とす):

```json
{
  "mode": "separate-window",
  "askEveryTime": false,
  "updatedAt": 1783440000000
}
```

- `mode`: `"same-window"` または `"separate-window"`
- `askEveryTime`: true のとき、参加操作ごとに開き方を選ぶ UI を出す
- account data event type は `in.selfmatrix.call_window_mode` を候補名とする
- localStorage の端末オーバーライドは account data と同じ値構造を使い、`mode` または `askEveryTime` の一部だけを上書きできるようにする
- account data 書き込みは既存値を読んでから merge する。古い snapshot で別設定を落とさないよう、`shellLayout.ts` の merge helper と同じ考え方を使う

### 参加経路ごとの UI

「すべての参加経路が設定に従う」は、次の意味で固定する。

| 参加経路 | askEveryTime = OFF | askEveryTime = ON |
| --- | --- | --- |
| 通話画面前の参加ボタン | `mode` に従って参加 | 分割ボタンまたは隣接メニューで「このウィンドウ / 別ウィンドウ」を選ぶ |
| チャンネル一覧・ヘッダー等のクイック参加 | `mode` に従って参加 | 小さな popover/menu で「このウィンドウ / 別ウィンドウ」を選ぶ。選ばれるまで参加しない |
| 着信応答 | `mode` に従って応答 | 応答 UI に 2 アクションを出す。UI 上どうしても出せない場合は既定 `mode` に従うが、その例外を実装メモに残す |

askEveryTime = ON の経路で、実装都合により選択 UI が出せない場合は「黙って同一ウィンドウで参加」しない。
既定 `mode` に従うか、参加を中止して明確なエラーを出す。

### 各操作の挙動

- **別窓を閉じる = 通話から退出** (「閉じたらメインに戻す」は再接続を強いるので採らない)
- **「このウィンドウに戻す」ボタンは残す** — ただし移動なので再接続 (~1.4 秒) を伴う。
  ツールチップ等で「再接続します」と明示する
- **メインウィンドウを閉じる/リロード = 通話窓も道連れで閉じる** — Matrix クライアント本体
  (認証・同期・widget との通信相手) がメインウィンドウに居るための原理的制約。受け入れる。
  実装では `pagehide` / `beforeunload` 相当で best-effort の `hangup()` を送ってから popup を閉じる。
  ブラウザ unload 中なので成功保証はしないが、単なる `popup.close()` だけに依存しない
- **途中ポップアウト (⧉) は併存** — 「このウィンドウで参加 → あとから出す」経路も従来どおり (再接続あり)
- **すべての参加経路が設定に従う** — 通話画面前の参加ボタン、チャンネル一覧からのクイック参加、着信応答
- **通話中に別チャンネルの通話へ移る** — 既存通話を先に退出してから次の通話へ参加する。
  同じウィンドウ名を再利用するので、通話窓は増えない。既存の CallPopout / iframe / LiveKit 接続を
  `hangup()` し、Close/HangupCall または timeout を待ってから次の `CallPopout` を生成する。
  退出が確認できない場合は二重参加を避けるため次の参加を中止し、エラーを出す
- **ポップアップブロック時は参加しない** — 開き方の設定が「別ウィンドウ」なのに黙って
  このウィンドウで開くのはユーザーの意図に反する (2026-07-07 運用者決定。当初案の
  「このウィンドウでフォールバック参加」を置換)。参加を中止し、
  「ポップアップがブロックされたため参加できませんでした。このサイトのポップアップを
  許可してください」という明確なエラーを表示する (設定画面への導線を添えてもよい)
- **メイン側バナーは参加開始直後から出す** — 別窓を開いた直後、EC が JoinCall を出す前でも
  「通話を別ウィンドウで開いています」を表示する。joined 後は既存の PoppedOutBanner 相当に切り替える。
  ポップアップブロックや参加失敗時はバナーを消してエラーを残す

### Discord 実機観察からの追加要件 (2026-07-07、運用者録画より)

運用者が Discord デスクトップ版のポップアウト挙動を録画・実演した (音声解説付き 2:16。
**動画そのものは非公開・運用者ローカル保管 — GitHub には置かない運用**)。観察された挙動:

- ポップアウト窓には**通話 UI が丸ごと**付いてくる: フォーカス表示 + ミニタイル列 (クリックで切替) +
  コントロールバー (ミュート/配信開始/退出) + 品質バッジ (720p 30FPS 等) + フルスクリーン切替
- ユーザー右クリックで音量スライダー・ミュート・「ユーザーをポップアウト」等のメニュー
- **ポップアウト窓から自分の配信を開始できる** (共有対象ピッカーも窓内で開く。画質選択付き)
- 既定では最前面ではないが、**ピン留めボタンで最前面固定にできる**
- 「Discord に戻る」ボタンで窓からメインへ戻れる (Discord は Electron アプリなので無再接続で戻る)
- 視聴映像は画質「ソース」なら窓サイズに追従

これを踏まえた本設計への反映:

1. **通話ウィンドウには EC 自身のフッター UI を表示する (必須)** — メイン埋め込みでは EC フッターを
   URL パラメータで隠し cinny 側のコントロールバーが代替しているが、**別ウィンドウには cinny の
   コントロールバーが存在しない**。widget URL の生成を別窓時だけ変え、EC 標準のフッター
   (ミュート / 画面共有 + 画質・FPS ピッカー / レイアウト切替 / 強調選択 / 退出) を窓内に出す。
   これで「窓から配信開始」「窓内で完結する操作」が Discord 同等になる
2. ミニタイル切替・音量メニュー・受信品質バッジは EC 実装済み機能でカバー済み
3. **最前面固定 (ピン留め) は通常の `window.open` では実現不可** (Web にその API が無い)。
   Chromium 系限定の Document Picture-in-Picture API なら常時最前面の窓を作れるが、制約が多いので
   **LATER (将来検討)** として記録。初期実装ではやらない
4. 「メインに戻す」が再接続を伴うのは Discord との**意図的な差分** (あちらはネイティブアプリ、
   こちらはブラウザの iframe 制約)。戻す操作を使わない運用 (最初から別窓) を既定にすることで実害を消す
5. フルスクリーン切替はブラウザ/EC 標準機能で可

### 補助機能との関係

- **ストリーム単体 🗗 (再接続なし)**: 通話ウィンドウからさらに別窓へ出せる (ウィンドウからの
  ウィンドウはユーザー操作起点なら開ける)。※保留中の「グリッドタイルへの 🗗 追加」タスクは
  この設計の確定を受けて再開できる
- E2EE・参加権限は通常参加と完全に同一 (参加する場所が違うだけで、参加処理そのもの)

### 窓のサイズ/位置の記憶 (スコープ内 — 2026-07-07 運用者決定で nice-to-have から昇格)

- 通話窓のサイズと位置を**端末ローカル (localStorage) に記憶**し、次回は同じ場所・同じ大きさで開く
  (現状は 960x640 固定)。マルチモニタ環境で毎回置き直す手間をなくす、Discord 的な快適さ
- 記憶タイミングは窓を閉じる/通話終了時 (resize/move の逐次保存でもよい、実装しやすい方)。
  保存した位置が現在の画面外になる場合 (モニタ構成変更) は既定位置にフォールバック
- `openCallPopoutWindow()` は固定 feature string ではなく、保存済み geometry を受け取って
  `left/top/width/height` を組み立てる。最小サイズを下回る保存値は破棄する
- ブラウザによって popup の位置取得・move イベントが不安定な場合は、サイズだけでも保存してよい。
  ただし「保存できなかった値」を壊れた値として次回に使わない

## 5. 実装メモ (リポジトリが見える実装者向け — 見えない読者は §4 までで検討可能)

- 中核: `cinny/src/app/hooks/useCallEmbed.ts` の `useCallStart` に分岐 — 開き方が「別ウィンドウ」なら
  `openCallPopoutWindow()` → `new CallPopout(...)` を直接生成 (既存 `useCallPopout` から
  hangup 待ちを除いた形)。`CallPopout` / `PoppedOutBanner` / popup 所有権ガードは実装済みを流用。
  新規参加時は cinny 側のみで完結、EC 側の変更は不要。ただし widget URL 生成時に別窓では EC フッターを
  隠さないパラメータへ切り替える
- 既に通話中の状態から別チャンネルへ移る場合は、直接 `new CallPopout(...)` せず、旧 embed の
  `hangup()` / Close 待ちを必ず挟む
- 設定の読み書きは `screenShareSettings.ts` (localStorage 契約) と `shellLayout.ts`
  (account data + マージヘルパー) のパターンを合成。account data のイベントタイプ名は実装時に確定
  (例: `in.selfmatrix.call_window_mode`)
- 分割ボタン UI は Prescreen の参加ボタン部。クイック参加・着信応答は小さな menu/popover か 2 アクション応答を用意する
- テスト:
  - 二層設定解決 (override 優先) の unit
  - account data merge で片方の設定を落とさない unit
  - 別窓で参加 → メインに接続中バナー → joined バナー
  - 初回別窓参加で HangupCall が発生しないこと
  - askEveryTime ON で Prescreen / quick join / incoming answer が同じ選択ルールになること
  - 別チャンネル通話へ移る時、旧通話 hangup 完了前に次の join を開始しないこと
  - popup block 時は参加しないこと
