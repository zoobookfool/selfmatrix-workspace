Discord 動作ログweb 実測版

SelfMatrix「UI 総点検 (Discord 実物突き合わせ)」の基準器。デザイン提案ではなく、Discord が どう見えて・どう動くかの観察記録。

方法論

  • 実測日: 2026-07-13、Discord web 版 (discord.com) を Chrome の実セッションで観察
  • 観察のみ: 非公開 API・DOM 改変・裏技的操作は行っていない。見えている挙動をそのまま記録
  • 秘匿: 実サーバー名・実ユーザー名・実ドメインは非掲載。再現図の名前は「User A」「Bot B」等のプレースホルダ
  • web / デスクトップ差: 異なる箇所はその都度明記 (例: §6 画面共有ピッカー、§7 ポップアウト)
  • アセット非使用: Discord のロゴ・アイコン画像・音は一切含まない。CSS・図形・絵文字のみで近似再現

§6 画面共有ピッカー (web)

Discord web の画面共有ピッカーは Discord 独自 UI ではなく、Chrome ネイティブの getDisplayMedia ピッカーそのまま — つまり SelfMatrix web と同一の制約を持つ。独自ピッカーはデスクトップ版のみ (feature-matrix.md 1.2「システム音声(全体ミックス)をどのソース 共有でも付与」の行と一致)。

タブ構成と音声トグルの既定 (実測)

§7 ポップアウト/PiP

宿題あり

観察

Chrome には documentPictureInPicture API がある。
配信者側の操作範囲では document PiP は一度も開かなかった。参加者右クリックの 「ユーザーをポップアウト」はスポットライト表示化であり、別ウィンドウ化は未確認。
宿題: 視聴側 (他人の映像配信を見ている状態) での PiP/ポップアウト挙動が未検証。native M3 「web は原理的に無再接続ポップアウト不可」という前提の確認はここが本丸 — 友達の実配信がある時に再観察する。

§8 SelfMatrixへの示唆

今回の実測と feature-matrix / backlog の突き合わせ。

観察示唆関連文書
参加者右クリックメニューにユーザー音量スライダーが直接埋め込まれている (§5) backlog P1 の参照実装が採れた。プロフィール/メッセージ直下、ミュート系チェックボックスの手前に配置する 構成をそのまま踏襲できる backlog.md P1「話者オーバーレイ右クリックからの ユーザー単位音量調整」
テキストチャンネルの bot コマンド使用履歴が「○○さんが #play を使用しました」の折りたたみ表示 (小さい行+本体メッセージ) 通話参加/終了イベントの洪水対策にも同じ折りたたみ表示パターンを転用できる backlog.md P2「後追い参加時に過去の通話参加/終了イベントが 洪水」(RoomTimeline.tsx)
bot 名の右に「✓アプリ」blurple ピルバッジ (§3) チャンネル/ユーザーの描き分けの具体的な参照点になる backlog.md P2「UI総点検 (Discord実物突き合わせ) + チャンネル/ユーザーの描き分け」
画面共有ピッカーは Chrome 標準 UI そのもの。web/native の差は既に文書化済みの制約と一致 (§6) feature-matrix 1.2 の記述を実物で裏付け。追加対応は不要 feature-matrix.md 1.2「システム音声をどのソース 共有でも付与」
視聴側 PiP/ポップアウト挙動が未検証 (§7) native M3 の「web は原理的に無再接続ポップアウト不可」前提の確認が残課題。友達の配信時に再観察する feature-matrix.md 1.2「無再接続ポップアウト」
